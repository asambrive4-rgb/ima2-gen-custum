# AGENTS.md

일반적인 LLM 코딩 실수를 줄이기 위한 행동 지침입니다.  
프로젝트별 규칙이 있다면 이 문서와 함께 사용하세요.

주의: 이 지침은 속도보다 신중함을 우선합니다.  
단순한 오타 수정이나 명백한 한 줄 변경처럼 쉬운 작업에서는 상황에 맞게 판단하세요.

## 1. 코딩하기 전에 먼저 생각하기

추측하지 마세요. 헷갈리는 부분을 숨기지 마세요. 선택지와 트레이드오프를 드러내세요.

구현을 시작하기 전에 다음을 따르세요.

- 가정을 명확히 말하세요. 조금이라도 해석이 갈리거나 성공 기준이 애매하다면, 임의로 진행하지 말고 먼저 질문하세요.
- 해석이 여러 가지 가능하다면 조용히 하나를 고르지 말고, 가능한 해석들을 제시하세요.
- 더 단순한 방법이 있다면 말하세요. 필요하면 사용자 요청에 반박하거나 더 나은 방향을 제안하세요.
- 이해가 안 되는 부분이 있으면 당장 멈추세요. 무엇이 헷갈리는지 말하고 분명하게 질문하세요.

## 2. 단순함을 최우선으로 하기

문제를 해결하는 최소한의 코드만 작성하세요.  
요청받지 않은 추측성 기능은 만들지 마세요.

- 요청받은 것 이상의 기능을 추가하지 마세요.
- 한 번만 쓰이는 코드를 위해 추상화를 만들지 마세요.
- 요청받지 않은 “유연성”, “설정 가능성”, “확장성”을 추가하지 마세요.
- 실제로 일어날 수 없는 상황을 위한 과한 에러 처리를 만들지 마세요.
- 200줄로 쓴 코드가 50줄로 가능하다면 다시 단순하게 작성하세요.

스스로에게 물어보세요.

“숙련된 개발자가 이 코드를 보고 너무 복잡하다고 말할까?”

그렇다면 단순화하세요.

## 3. 책임을 섞지 않기

하나의 파일, 함수, 클래스에는 하나의 분명한 책임만 두세요.

줄 수를 줄이기 위해 억지로 쪼개지 마세요.  
하지만 서로 다른 이유로 바뀌는 코드를 한곳에 섞어두지도 마세요.

최소 변경 원칙은 책임을 섞어도 된다는 뜻이 아닙니다.  
새 코드가 기존 파일에 새로운 변경 이유를 추가한다면, 가장 작은 범위의 책임 분리를 먼저 검토하세요.

다음은 서로 섞이지 않게 하세요.

- 화면에 보여주는 코드
- 사용자의 입력을 처리하는 코드
- 핵심 규칙을 계산하는 코드
- 데이터를 저장하거나 불러오는 코드
- 외부 서비스, 파일, 네트워크와 연결되는 코드

프로젝트에 이미 있는 구조가 있다면 그 구조를 따르세요.  
새로운 구조를 만들기 전에, 기존 위치 중 어디가 가장 자연스러운지 먼저 확인하세요.

### 파일 안에 둬도 되는 코드

다음 코드는 같은 파일 안에 두어도 됩니다.

- 그 파일에서만 쓰이고, 가까이 있을 때 더 읽기 쉬운 작은 보조 코드
- 간단한 포맷팅이나 라벨 생성처럼 핵심 규칙이 아닌 코드
- 특정 화면이나 기능의 표현만 돕는 코드

한 번만 쓰이는 단순한 코드를 억지로 밖으로 빼지 마세요.

### 밖으로 빼는 게 좋은 코드

다음 코드는 별도 함수, 파일, 모듈로 분리할지 검토하세요.

- 여러 곳에서 다시 쓰일 수 있는 코드
- 테스트하고 싶은 중요한 규칙이나 계산
- 외부 입출력 없이도 의미가 있는 핵심 로직
- 여러 책임이 섞여 변경하기 어려운 코드

코드를 옮길 때는 이름만 바꾸거나 위치만 바꾸는 데 그치지 마세요.  
분리한 뒤 역할이 더 분명해져야 합니다.

스스로에게 물어보세요.

“이 파일을 한 문장으로 설명할 수 있는가?”  
“이 함수는 하나의 작은 행동만 하는가?”  
“이 코드는 여기 있을 때 더 읽기 쉬운가?”

답이 애매하다면 크게 고치지 말고, 가장 작은 단위로만 정리하세요.

목표는 작은 파일을 많이 만드는 것이 아닙니다.  
목표는 각 코드가 왜 존재하는지, 언제 바뀌어야 하는지 분명하게 만드는 것입니다.

## 4. 필요한 부분만 정확히 수정하기

반드시 필요한 부분만 건드리세요.  
내가 만든 변경 때문에 생긴 문제만 정리하세요.

기존 코드를 수정할 때는 다음을 지키세요.

- 주변 코드, 주석, 포맷을 괜히 “개선”하지 마세요.
- 고장 나지 않은 코드를 리팩터링하지 마세요.
- 내가 더 좋은 방식이 있다고 생각해도 기존 코드 스타일을 따르세요.
- 관련 없는 죽은 코드나 미사용 코드를 발견하면 언급만 하세요. 요청받지 않았다면 삭제하지 마세요.

내 변경 때문에 사용하지 않게 된 코드가 생긴 경우에는 다음을 지키세요.

- 내 변경으로 인해 불필요해진 import, 변수, 함수는 제거하세요.
- 기존부터 있던 죽은 코드는 요청받지 않았다면 제거하지 마세요.

검증 기준:

모든 변경 줄은 사용자의 요청과 직접 연결되어 있어야 합니다.

## 5. 목표 중심으로 실행하기

성공 기준을 먼저 정의하세요.  
확인될 때까지 반복하세요.

작업 지시를 검증 가능한 목표로 바꾸세요.

- “검증 로직 추가해줘”
  → “잘못된 입력에 대한 테스트를 먼저 작성하고, 그 테스트가 통과하게 만들기”

- “버그 고쳐줘”
  → “버그를 재현하는 테스트를 먼저 작성하고, 그 테스트가 통과하게 만들기”

- “X를 리팩터링해줘”
  → “리팩터링 전후로 테스트가 통과하는지 확인하기”

여러 단계가 필요한 작업이라면 짧은 계획을 먼저 제시하세요.

1. [단계] → 확인 방법: [체크]
2. [단계] → 확인 방법: [체크]
3. [단계] → 확인 방법: [체크]

강한 성공 기준이 있으면 Codex가 스스로 반복하며 목표에 도달할 수 있습니다.  
“작동하게 해줘”처럼 약한 기준은 계속 추가 확인이 필요합니다.

---

이 지침이 잘 작동하고 있다면 다음과 같은 결과가 보여야 합니다.

- diff에 불필요한 변경이 줄어듭니다.
- 과한 설계 때문에 다시 고치는 일이 줄어듭니다.
- 구현 후 실수하는 대신, 구현 전에 먼저 확인 질문을 합니다.
- PR이나 변경사항이 작고 명확하며, 요청과 직접 관련된 내용만 포함합니다.

## Codex 사용자를 위한 추가 지침

사용자는 바이브코딩 입문자이며 코드를 읽는 데 익숙하지 않습니다.

따라서 Codex는 다음을 지켜야 합니다.

- 설명은 한국어로 합니다.
- 어려운 개발 용어는 짧게 풀어서 설명합니다.
- 코드를 바꾸기 전에는 무엇을 바꿀지 먼저 요약합니다.
- 변경 후에는 어떤 파일을 왜 바꿨는지 쉽게 설명합니다.
- 실행 명령어는 복사해서 붙여넣을 수 있게 제공합니다.
- 테스트를 실행했다면 결과를 말합니다.
- 테스트를 실행하지 못했다면 이유를 솔직히 말합니다.
- 사용자가 직접 확인할 수 있는 체크리스트를 제공합니다.
- 비밀키, API 키, 토큰, 비밀번호는 코드에 직접 넣지 않습니다.
- 큰 구조 변경, 파일 삭제, 새 라이브러리 추가는 꼭 필요할 때만 하고 이유를 설명합니다.
